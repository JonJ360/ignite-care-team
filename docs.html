Technical Notes
Stack, patterns, conventions
›
Stack Expand ▶
Pattern A build: single-file plain HTML + vanilla JS, no build step. Supabase JS v2 via jsDelivr CDN. Google Fonts (Oswald + Inter in the app; this doc uses the standard doc set). Hosted on GitHub Pages as index.html + docs.html.
BAKED_CONFIG block Expand ▶
Same convention as the Time Tracker / AR CRM: a BAKED_CONFIG block near the top of the script holds SUPABASE_URL and SUPABASE_ANON_KEY. Every delivered version ships with these baked once set. Currently empty pending project setup. With config empty, the form renders but submit shows a not-connected notice and staff login is disabled.
Versioning convention Expand ▶
Source lives as care-team_v1_XX.html in the working directory. Every change bumps the version and delivers index.html + docs.html together, with What's New and Technical Notes updated each version.
Workflow model (V1.08) Expand ▶
The unit of work is the assigned case. Assignment puts it on that member's Dashboard and My Cases. Logging an activity stamps last_activity_at and sets next_followup_date — that date drives "Follow-ups due" and the overdue report. The follow-up field prefills with the case’s existing follow-up date; leaving it blank when logging clears the scheduled follow-up (no next step planned). Cards surface "No contact yet" and "14+ days since contact" flags on open cases. Write rule: owner-or-admin on case fields, admin-only reassignment, everyone can log activity on any case (covering a teammate still gets recorded).
Staff lifecycle — create-staff Edge Function V2 Expand ▶
Auth-user operations require the service role key, which never ships in a public page. The function (name unchanged: create-staff) is multi-action as of V2: create, reset_password, update (name/email, auth kept in sync), and delete. Every action verifies the caller's session token and requires staff_profiles.role = 'admin' server-side. Delete guards against self-removal and releases references first (requests/tasks unassigned) so the profile row can drop; routing rows cascade, note authorship sets null. Source ships as create-staff.ts. Own-password changes use sb.auth.updateUser client-side.
Case notes Expand ▶
New case_notes table, loaded lazily when a request's detail panel opens (newest first). RLS: all staff read; insert only as yourself (author_id = auth.uid() enforced in policy); no updates or deletes by staff — the log is append-only for integrity, with admin-only delete for cleanup. Deleting a request cascades its notes; removing a staff member nulls their authorship, displayed as "Former staff."
Views & state Expand ▶
Four views in one file: public form, success, staff login, portal (tabs: Requests / Statistics / Tasks / Team & Routing). All portal data loads in one parallel fetch on login. Statistics are computed client-side from the requests array — no extra queries. Assigning a New request auto-bumps status to Assigned. Admin tabs are hidden for staff role in the UI and blocked at the database by RLS.
Need areas & statuses Expand ▶
Eight need areas (stored verbatim as text): Miscarriage · Stillbirth · Infertility · In vitro fertilization (IVF) · Postpartum anxiety or depression · Child born with special needs · Serious child medical / health issues · Other tragedy or trauma. Statuses: new → assigned → contacted → walking → closed (displayed as New, Assigned, Contacted, Walking Alongside, Closed).

Data Flow
QR scan to closed case
›
1
Scan. Person scans the QR code and opens the intake form.
2
Submit. Form validates and inserts a row into care_requests with status New using the anon key.
3
Review. Admin opens the Requests tab; New requests carry the gold badge.
4
Route. The detail panel shows the routing suggestion — team members mapped to the request's primary need.
5
Assign. Admin assigns a team member; status auto-bumps to Assigned.
6
Walk. Status progresses: Contacted → Walking Alongside as the team member connects and journeys with the family.
7
Close. Case marked Closed. Statistics update live from the same data.

V1.08 migration SQL — run once (existing projects)

-- Structured activity log (existing notes become 'note' type)
alter table case_notes add column activity_type text not null default 'note'
  check (activity_type in ('call','text','email','in_person','meeting','prayer','note'));
alter table case_notes add column outcome text
  check (outcome in ('connected','left_message','no_answer','meeting_scheduled'));

-- Workflow fields on requests
alter table care_requests add column next_followup_date date;
alter table care_requests add column last_activity_at timestamptz;

-- Write-access rule: staff update only their own cases; admin anything.
drop policy "staff update" on care_requests;
create policy "update own or admin" on care_requests
  for update to authenticated
  using (assigned_to = auth.uid() or is_admin())
  with check (assigned_to = auth.uid() or is_admin());

-- Tasks: staff create their own, work what's theirs; admin everything.
drop policy "admin write tasks" on tasks;
create policy "create own tasks" on tasks
  for insert to authenticated with check (created_by = auth.uid());
create policy "update own or admin tasks" on tasks
  for update to authenticated
  using (assigned_to = auth.uid() or created_by = auth.uid() or is_admin())
  with check (assigned_to = auth.uid() or created_by = auth.uid() or is_admin());
create policy "delete own or admin tasks" on tasks
  for delete to authenticated using (created_by = auth.uid() or is_admin());

V1.06 migration SQL — run once (existing projects)

create table case_notes (
  id uuid primary key default gen_random_uuid(),
  created_at timestamptz not null default now(),
  request_id uuid not null references care_requests(id) on delete cascade,
  author_id uuid references staff_profiles(id) on delete set null,
  body text not null
);

alter table case_notes enable row level security;

create policy "staff read notes" on case_notes
  for select to authenticated using (true);
create policy "staff add own notes" on case_notes
  for insert to authenticated with check (author_id = auth.uid());
create policy "admin delete notes" on case_notes
  for delete to authenticated using (is_admin());

-- Admin-only request deletion (no delete policy existed before, by design)
create policy "admin delete requests" on care_requests
  for delete to authenticated using (is_admin());

Full SQL — run once in the Supabase SQL Editor

-- ========== TABLES ==========
create table staff_profiles (
  id uuid primary key references auth.users(id) on delete cascade,
  full_name text not null,
  email text not null,
  role text not null default 'staff' check (role in ('admin','staff')),
  created_at timestamptz not null default now()
);

create table care_requests (
  id uuid primary key default gen_random_uuid(),
  created_at timestamptz not null default now(),
  first_name text not null,
  last_name text not null,
  gender text check (gender in ('female','male','unspecified')),
  relationship_status text not null check (relationship_status in ('single','married','dating')),
  partner_first_name text,
  partner_last_name text,
  email text not null,
  phone text,
  preferred_contact text check (preferred_contact in ('email','call','text')),
  home_church boolean not null,
  life_group boolean not null,
  needs text[] not null,
  primary_need text not null,
  details text,
  status text not null default 'new' check (status in ('new','assigned','contacted','walking','closed')),
  assigned_to uuid references staff_profiles(id)
);

create table need_routing (
  id uuid primary key default gen_random_uuid(),
  need_area text not null,
  staff_id uuid not null references staff_profiles(id) on delete cascade,
  unique (need_area, staff_id)
);

create table tasks (
  id uuid primary key default gen_random_uuid(),
  created_at timestamptz not null default now(),
  title text not null,
  description text,
  assigned_to uuid references staff_profiles(id),
  request_id uuid references care_requests(id) on delete set null,
  due_date date,
  status text not null default 'open' check (status in ('open','done')),
  created_by uuid references staff_profiles(id)
);

-- ========== ADMIN CHECK (security definer avoids RLS recursion) ==========
create or replace function is_admin() returns boolean
language sql security definer set search_path = public stable as $$
  select exists (
    select 1 from staff_profiles
    where id = auth.uid() and role = 'admin'
  );
$$;

-- ========== ROW LEVEL SECURITY ==========
alter table care_requests  enable row level security;
alter table staff_profiles enable row level security;
alter table need_routing   enable row level security;
alter table tasks          enable row level security;

-- care_requests: public may INSERT only; all staff read/update
create policy "public insert" on care_requests
  for insert to anon, authenticated with check (true);
create policy "staff read" on care_requests
  for select to authenticated using (true);
create policy "staff update" on care_requests
  for update to authenticated using (true) with check (true);

-- staff_profiles: authenticated read; admin manages
create policy "staff read profiles" on staff_profiles
  for select to authenticated using (true);
create policy "admin insert profiles" on staff_profiles
  for insert to authenticated with check (is_admin());
create policy "admin update profiles" on staff_profiles
  for update to authenticated using (is_admin()) with check (is_admin());
create policy "admin delete profiles" on staff_profiles
  for delete to authenticated using (is_admin());

-- need_routing: authenticated read; admin manages
create policy "staff read routing" on need_routing
  for select to authenticated using (true);
create policy "admin write routing" on need_routing
  for all to authenticated using (is_admin()) with check (is_admin());

-- tasks: authenticated read; admin manages
create policy "staff read tasks" on tasks
  for select to authenticated using (true);
create policy "admin write tasks" on tasks
  for all to authenticated using (is_admin()) with check (is_admin());

Security Notes
What protects this data
›
⚠️ Do not deploy the anon key before running the RLS script. With RLS off, the anon key exposes every row publicly. This app stores pastoral-care data — pregnancy loss, postpartum mental health, child medical issues. Treat it accordingly.
Insert-only public access. The anon key can write a new request and nothing else — no reads, no updates, no other tables.
Admin writes enforced in Postgres. Roles, routing, and tasks require is_admin() (security-definer function) — hiding the tabs in the UI is convenience, the database is the gate.
No credentials in the codebase. Only the Supabase URL and anon key are baked. Passwords live exclusively in Supabase Auth. Visibility vs. write access: all staff read everything; case edits are owner-or-admin and reassignment admin-only (RLS-enforced as of V1.08).
